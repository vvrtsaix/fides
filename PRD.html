Loyalty Engine

An enterprise-grade, headless loyalty platform that treats points like financial transactions — built on absolute data integrity, velocity, and a clean separation of concerns.

Overview & Architectural Principles

This document establishes the product and functional requirements for an enterprise-ready, headless loyalty engine. The platform treats loyalty data similarly to financial transactions, prioritizing absolute data integrity, performance velocity, and a clean separation of concerns.

Core Philosophy

  • Headless & API-First. The engine exposes transactional data and rule structures purely via REST APIs and webhooks. It relies entirely on external frontends (Web, Mobile, POS) for the presentation layer.
  • Single Responsibility. The platform explicitly manages rules, point computations, financial ledgers, and reward tokens. It does not store product SKUs, marketing copy, or full customer biometric profiles.
  • Immutable Ledger Model. Point values are never altered via arbitrary addition/subtraction database commands. All balance modifications must occur through append-only accounting transactions.

Module-by-Module Functional Requirements

2.1Identity & Cache Module

Maps external users to the internal loyalty ecosystem and powers the real-time, high-concurrency presentation layer.

  • FR-1.1 External Identity Mapping

    The system must accept an external_id string from third-party systems (CRMs, Shopify, POS, or ERPs) to associate active loyalty behaviors without duplicating full customer profiles.

  • FR-1.2 Atomic Cache Maintenance

    The system must maintain a denormalized cache table (customer_balances). Every write to the ledger must recalculate and update spendable_balance, locked_balance, lifetime_earned, and lifetime_redeemed inside the same transaction boundary.

  • FR-1.3 Dynamic Tier Evaluation

    Upon any change to lifetime_points, the engine must cross-reference thresholds in the tiers table and instantly update the customer's current_tier_id if a milestone is breached.

2.2Event Ingestion & Rules Engine

Ingests unstructured user events asynchronously and converts them into point actions based on marketing conditions.

  • FR-2.1 Schema-less Event Intake

    The loyalty_events pipeline must ingest flexible JSONB payloads for seamless compatibility across integrations (e.g. cart arrays from e-commerce vs. location coordinates from a mobile app).

  • FR-2.2 Asynchronous Ingestion Status

    Events must be written immediately with an initial state of PENDING. An async worker evaluates them against rules, shifting state to PROCESSED or FAILED.

  • FR-2.3 Conditional Processing Engine

    The engine must match incoming event_type strings to active rows in earning_rules, evaluating the rule's JSONB condition block against event parameters. If true, base points and modifiers are minted.

2.3Financial Ledger & Lock Management

Handles data integrity, anti-fraud isolation, and append-only ledger processing.

  • FR-3.1 Immutable Ledger Modifications

    Points can only be adjusted via five strict transaction types: EARN, REDEEM, EXPIRE, ADJUSTMENT, and UNLOCK.

  • FR-3.2 FEFO Point Expiration Tracking

    The system must track expires_at timestamps at the individual transaction-row level. Redemptions consume points from the oldest unexpired earning rows first — First-Expiring, First-Out — via the available_amount ledger track.

  • FR-3.3 Transactional Fraud Holds

    To support return windows or pending checkouts, the engine must support isolation via points_locks. Locked points are deducted from spendable_balance and added to locked_balance until checkout completes (FULFILLED) or times out (RELEASED).

2.4Campaigns, Rewards & Fulfillments

Acts as the transactional bridge for reward inventory management and unique asset distribution.

  • FR-4.1 Budget Overdraft Protection

    Before any reward is issued, the engine must verify that campaigns.current_spend plus the reward value does not exceed the defined budget_cap.

  • FR-4.2 Race Condition Inventory Safeguards

    Reward stock (available_stock) must use pessimistic database locking during checkout so two users cannot claim a single remaining voucher in the same millisecond.

  • FR-4.3 Unique Asset Minting

    A successful redemption must deduct points via the ledger and mint a unique alphanumeric code into the vouchers table, flagged ISSUED, tracked until it transitions to USED or passes its valid_until deadline.

2.5Webhooks & System Audit Integration

Handles out-of-the-box communication with external downstream systems and protects security configurations.

  • FR-5.1 Outbound Event Messaging

    The system must monitor core updates and fire real-time, async HTTP POST requests to URLs registered in webhook_subscriptions based on occurrences (e.g. customer.tier_upgraded).

  • FR-5.2 Exponential Webhook Backoff

    Failed outbound webhooks (recorded in webhook_logs) must auto-retry up to a hard configuration limit using exponential backoff before flagging the log entry as failed.

  • FR-5.3 Security Auditing

    Every operational config alteration (e.g. updating an earning-rule threshold, manually adjusting points) must log a snapshot of the change (old_values vs. new_values) into the read-only audit_logs table.

Non-Functional Requirements

3.1 — Data Integrity & Concurrency

  • ACID Compliance. All operations binding points_ledger, points_locks, and customer_balances must share strict atomic transactional safety.
  • Isolation Levels. The database connection must enforce repeatable-read isolation when modifying user balances to prevent phantom writes under high traffic.

3.2 — Performance Thresholds

<50ms
Read Latency

Requests targeting cached profile objects (customer_balances) via optimized index structures.

2,000/s
Write Processing

Background ingestion workers must process this many async event entries per second without cascading table-lock blockages.

3.3 — Security & Compliance

§

Data Minimization Notice. By design, the core system does not store passwords, full names, addresses, or payment-card tokens — heavily reducing the threat landscape for security teams.

  • Anonymization Actions. To comply with global privacy frameworks (GDPR/CCPA), the system must support an ANONYMIZED status toggle on the customer table — zeroing out identity strings (emails/phones) while retaining historical transaction lines to preserve financial accounting integrity.
